# tokmd 1.13 Syntax Evidence Dogfood

## Purpose

This note records a release-candidate dogfood run for the syntax-aware evidence
packet lane. It is a maintainer evidence note, not the final 1.13 release note.

The goal was to check whether the current packet flow:

```text
tokmd analyze
tokmd context
tokmd syntax
tokmd evidence-packet
```

produces a useful first-read order for high-risk review without claiming to
prove reachability, safety, undefined behavior, or merge readiness.

## Run Context

- Date: 2026-06-11
- Swarm commit: `c142505b`
- Candidate binary: workspace build with `--features ast`
- Workspace package version before release prep: `1.12.0`
- Ref window used for validation: `HEAD^` to `HEAD`
- Artifact root: `target/dogfood/v1.13-syntax/`

The fixture-backed TypeScript and Python cases are representative syntax
samples because this repository does not otherwise carry production TypeScript
or Python sources. The Rust real-parser case uses tokmd source code.

## Commands

Each case used the same packet shape:

```bash
cargo run -p tokmd --features ast -- analyze \
  --preset bun-ub \
  --format md \
  --effort-base-ref HEAD^ \
  --effort-head-ref HEAD \
  --no-progress \
  --output-dir <case>/sensors/tokmd \
  <path>

cargo run -p tokmd --features ast -- analyze \
  --preset bun-ub \
  --format json \
  --effort-base-ref HEAD^ \
  --effort-head-ref HEAD \
  --no-progress \
  --output-dir <case>/sensors/tokmd \
  <path>

cargo run -p tokmd --features ast -- context \
  --budget 64000 \
  --output <case>/sensors/tokmd/context.md \
  --force \
  <path>

cargo run -p tokmd --features ast -- syntax --no-progress <path> \
  > <case>/sensors/tokmd/syntax.json

cargo run -p tokmd --features ast -- evidence-packet \
  --preset bun-ub \
  --base HEAD^ \
  --head HEAD \
  --output <case>/sensors/tokmd/manifest.json \
  --analyze-md <case>/sensors/tokmd/analyze.md \
  --analyze-json <case>/sensors/tokmd/analyze.json \
  --context-md <case>/sensors/tokmd/context.md \
  --syntax-json <case>/sensors/tokmd/syntax.json \
  <path>
```

On Windows PowerShell, the first run used `>` redirection for `analyze.json`.
That produced UTF-16LE output, and `tokmd evidence-packet` rejected it with:

```text
failed to read analyze_json ... stream did not contain valid UTF-8
```

The passing run used tokmd file-output options where available and an explicit
UTF-8 writer for `syntax.json`. This is useful release friction: Windows users
should prefer tokmd output flags or explicit UTF-8 file writes when producing
packet artifacts.

## First-Read Examples

The Rust panic-seam fixture produced a useful review order:

```text
1. native_boundary/use: std::{convert::TryFrom, ffi::CString}
2. panic_seam/assert_macro: assert!(!ptr.is_null())
3. panic_seam/capacity_allocation: Vec::with_capacity
4. panic_seam/capacity_allocation: bytes.reserve
5. panic_seam/expect: raw.expect
6. panic_seam/fallible_conversion_expect: usize::try_from(requested).expect
7. panic_seam/indexing: bytes[index]
8. panic_seam/panic_macro: panic!("index outside packet")
9. panic_seam/unwrap: CString::new(raw).unwrap
10. fallible_conversion/fallible_conversion: usize::try_from
```

The TypeScript fixture also produced an actionable first-read list:

```text
1. native_boundary/call: (value as NativePointer).run
2. native_boundary/call: dlopen
3. native_boundary/call: nativeBinding.symbols.run!
9. native_boundary/static: bun:ffi
10. dynamic_execution/call: import
```

The Python fixture surfaced the highest-risk dynamic and native-boundary seams:

```text
1. dynamic_execution/dynamic_eval: eval
2. native_boundary/call: NativeBridge
3. native_boundary/call: ctypes.CDLL
4. native_boundary/from_import: ctypes
5. native_boundary/import: ctypes
```

`subprocess.run`, file I/O, exception handling, and dynamic import were present
in `syntax.json`, but did not reach the capped top 10 because native-boundary
hints crowded the ranking.

## Findings

- The packet contract works end to end with optional syntax evidence: each case
  produced `analyze.md`, `analyze.json`, `context.md`, `syntax.json`, and
  `manifest.json`.
- Syntax review priority reduces search space for fixture-shaped Rust,
  TypeScript, and Python native-boundary cases.
- The manifest is the right first file for bots and agents: it records artifact
  paths, status, non-claims, reproduction commands, and ranked review hints.
- The real tokmd Rust parser file showed ranking noise: unit-test assertions in
  parser source ranked as panic seams. This is not a parser failure, but it is a
  signal-quality follow-up.
- Windows PowerShell `>` redirection can create non-UTF-8 JSON artifacts. The
  packet verifier catches this, but the docs should keep steering operators
  toward tokmd output flags or explicit UTF-8 writes.

## Follow-Ups

- Consider category diversity or per-category caps in `review_priority` so
  native-boundary hints do not crowd out subprocess, dynamic import, file I/O,
  or exception seams.
- Consider marking test-only syntax regions or test modules so unit-test
  assertions do not dominate release-review packets.
- Keep full public-reachability panic-seam receipts as a post-1.13 lane unless
  a narrow advisory slice is explicitly assigned.

## Non-Claims

This dogfood run does not claim:

- `tokmd` proves undefined behavior exists or is absent;
- syntax review priority proves public reachability;
- fixture-backed TypeScript and Python samples replace a real Bun repository
  dogfood run;
- a `complete` packet is a merge verdict.
